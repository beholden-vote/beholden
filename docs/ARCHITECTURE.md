# Beholden — Architecture (Free-Tier Launch Edition)

**Design goal:** launch at **$0/month infrastructure cost** without compromising the PRD's
performance (map TTI <2s, dossier <300ms) or provenance guarantees — then scale up by
*adding* components, never rewriting.

The trick that makes $0 possible: **Beholden v1 has no runtime backend at all.**
Every dossier, style feed, pin list, graph neighborhood, and search index is pre-built
by the nightly pipeline and published as static files behind a free CDN. The "database"
at serve time is the CDN. Postgres enters the picture only when the public API ships (P1).

---

## 1. Component map (what runs where, and why it's free)

| # | Component | Technology | Host | Free tier reality check |
|---|-----------|-----------|------|------------------------|
| 1 | Code, issues, CI/CD | **GitHub public monorepo** | GitHub | Public repos get **unlimited** Actions minutes. Public code also *is* the transparency mission. |
| 2 | ETL orchestration | **GitHub Actions cron** (nightly + intra-day PTR/vote checks) | GitHub | Free. 6h job limit is far beyond our needs. Retries via workflow re-run; backfills via manual dispatch. |
| 3 | Transform engine | **DuckDB + Python** inside Actions runners | GitHub | Free, in-process, zero infra. Same SQL migrates to Postgres later. |
| 4 | Raw lake (immutable) | **Cloudflare R2**, versioned bucket | Cloudflare | 10 GB storage, 1M class-A + 10M class-B ops/mo free, **zero egress fees**. Raw filings/PDFs/CSV snapshots live here. |
| 5 | Published data (dossiers, feeds, graph, search index) | Static JSON on **Cloudflare Pages** (small files) + R2 (bulk) | Cloudflare | Pages: unlimited requests + bandwidth, 500 builds/mo, 20k files/25MB-per-file per deploy. ~8k dossier JSONs fit comfortably. |
| 6 | Map tiles | **PMTiles archives on R2**, custom domain through Cloudflare CDN | Cloudflare | Single-file archives read via HTTP range requests directly by the browser — **no tile server exists**. R2 behind your own domain = free egress + CDN cache. |
| 7 | Frontend | **Vite + React + MapLibre GL + deck.gl + pmtiles** (fully static SPA) | Cloudflare Pages | No SSR runtime = nothing to pay for and nothing to fall over on launch day. |
| 8 | Address → district | **Census Bureau geocoder** (free public API) → lat/lng → client-side `queryRenderedFeatures` against the district tiles | Census + client | Zero backend. The tiles themselves are the point-in-polygon index. |
| 9 | Search (officials/bills) | Pre-built **MiniSearch/Pagefind index** shipped as static chunks, queried client-side | Cloudflare Pages | ~8k officials ≈ a few-MB compressed index. Semantic/pgvector search deferred to P1. |
| 10 | Analytics | **Cloudflare Web Analytics** | Cloudflare | Free, no cookies, no individual tracking — matches the privacy stance. |
| 11 | Status/coverage dashboard | Static page generated by the pipeline (it's just more JSON) | Cloudflare Pages | Free by construction. |

**Explicitly NOT in v1 free architecture:** runtime Postgres, tile servers, Kubernetes,
Temporal, graph databases, vendor trade feeds, self-hosted analytics. Each has a defined
entry point in §5.

## 2. The only non-free line items (be honest about them)

| Item | Cost | Mitigation |
|---|---|---|
| Domain name | ~$10/yr | Unavoidable; Cloudflare Registrar at-cost. |
| LLM calls ("Known for" drafting, scanned-PDF extraction) | Pennies→dollars per run | Cache aggressively (regenerate only on input-hash change); run extraction only on new filings; or defer "Known for" to launch+1 and ship committee/leadership facts verbatim. |
| Commercial PTR feed (Quiver/FMP/Finnhub) | $$$/mo | **Skip in free edition.** Parse House Clerk + Senate eFD ourselves (E3-2/3/4) and seed/validate against free community datasets on GitHub (e.g., the senate/house stock-watcher data dumps). Dual-run discipline is preserved — the second source is just free. This resolves Open Question O1 as "internal-first." |

## 3. Data flow (nightly)

```
GitHub Actions (cron 06:00 UTC)
 ├─ fetch: congress.gov API, unitedstates/congress-legislators, Voteview CSV,
 │         OpenStates bulk, FEC API, Clerk/eFD indexes, TIGER (on redistricting change)
 ├─ land:  raw snapshots → R2 /raw/{source}/{date}/…       (immutable, versioned)
 ├─ transform: DuckDB runs db/migrations DDL + dbt-style SQL models, quality gates
 │             (spine resolution ≥99.5% or the job FAILS — nothing partial publishes)
 ├─ build: dossiers/*.json · stylefeeds/*.json · pins/*.json · graph/*.json ·
 │         search-index/* · coverage.json
 └─ publish: small files → Pages deploy · bulk + PMTiles → R2 · git tag = pipeline_version
```

Provenance enforcement lives in the build step: the dossier builder validates every
section's provenance envelope against `data-contracts` schemas and refuses to publish
violations. CI runs the same validation on PRs that touch builders.

## 5. Scale-up path (add, don't rewrite)

| Trigger | Add | Notes |
|---|---|---|
| Public API (P1) ships | **Neon or Supabase Postgres** (both have free tiers with pgvector) loaded from the same DuckDB models; thin read API on **Cloudflare Workers** (100k req/day free) | Contracts are already the API — the static JSON and the API serve identical documents. |
| Semantic search demand | **Cloudflare Vectorize** or pgvector on the P1 Postgres | Embeddings already computed in pipeline. |
| Actions cron too coarse / complex backfills | **Temporal Cloud** or self-hosted worker | Adapters already isolate sources; orchestration swaps cleanly. |
| Traffic makes Pages file limits awkward | Move all dossiers to R2 behind the same domain | Path structure unchanged; clients never notice. |
| Local officials (Phase 3) | Licensed feed adapter + `place`/`county` divisions | Schema already accepts arbitrary OCD depth. |

## 6. Why this is still "bleeding edge"

Static-first is not a compromise here — it is the current best practice for exactly this
shape of product: PMTiles (single-file tile archives over range requests) replaced tile
servers; DuckDB-in-CI replaced warehouse clusters for sub-10GB analytical workloads;
CDN-published JSON with content hashing replaces read APIs for data that changes daily,
not per-request. The result is a system with **zero servers to attack, zero 3am pages,
and a public, reproducible pipeline** — which for a transparency product is itself a
credibility feature: anyone can audit exactly how every published fact was produced.
